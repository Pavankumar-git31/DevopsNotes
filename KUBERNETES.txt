Pavan, this is a very important Kubernetes scheduling concept used in real production clusters. It explains why pods sometimes fill one node first and sometimes spread across nodes.

This behavior mainly depends on scheduler strategies.

The two important strategies are:

1. Bin Packing
2. Pod Spreading

Think of it like packing clothes in suitcases.

1️⃣ Bin Packing (Fill one node first)

Bin packing means scheduler tries to fill one node as much as possible before using other nodes.

Example cluster:

Node1
Node2
Node3

Pod requirement:

CPU request = 1

Node capacity:

Node CPU = 4

Pods arriving:

4 pods

Scheduler result with bin packing:

Node1 → 4 pods
Node2 → 0
Node3 → 0

So one node is fully used first.

Why Kubernetes does this?

Because companies want to reduce infrastructure cost.

Example:

3 nodes running = expensive
1 node running = cheaper

So if nodes become empty:

Cluster autoscaler removes unused nodes

This saves cloud cost.

This strategy is used when:

Goal = Cost optimization
2️⃣ Pod Spreading (High Availability)

Sometimes Kubernetes spreads pods across nodes.

Example cluster:

Node1
Node2
Node3

Deployment:

3 replicas

Scheduler result:

Node1 → 1 pod
Node2 → 1 pod
Node3 → 1 pod

Why?

Because if Node1 crashes, other pods still run.

Example bad case:

Node1 → 3 pods
Node2 → 0
Node3 → 0

If Node1 fails:

All pods gone
Application down

So spreading improves fault tolerance.

3️⃣ Kubernetes default behavior

Kubernetes tries to balance both:

Cost optimization
+
High availability

Great Pavan 👍 Now let’s understand how the Kubernetes Scheduler actually chooses a node for a pod.
This is a very common DevOps interview topic and also important for system design.

The scheduler works in 3 main phases:

1. Filtering
2. Scoring
3. Binding

Think of it like a company hiring process.

Job applicants → Screening → Ranking → Hiring

Pods are the job applicants, and nodes are the companies.

1️⃣ Phase 1 — Filtering (Eliminating unsuitable nodes)

First, the scheduler checks which nodes are capable of running the pod.

It removes nodes that cannot satisfy the pod requirements.

Example cluster:

Node1
Node2
Node3
Node4

Pod requirements:

CPU request = 2
Memory request = 2GB
zone = us-east-1a

Filtering checks:

• CPU availability
• Memory availability
• Node selector
• Node affinity
• Taints / tolerations
• Volume requirements
• Max pods per node

Example result:

Node1 → Not enough CPU ❌
Node2 → Wrong zone ❌
Node3 → Tainted node ❌
Node4 → Valid node ✅

After filtering:

Candidate nodes = Node4

If no nodes pass filtering, the pod becomes:

Pending

This is when Cluster Autoscaler may create new nodes.

2️⃣ Phase 2 — Scoring (Choosing the best node)

If multiple nodes pass filtering, Kubernetes scores them to find the best node.

Example filtered nodes:

Node1
Node2
Node3

Scheduler assigns scores based on different rules.

Important scoring factors:

1. Resource balance

Kubernetes tries to balance CPU and memory usage across nodes.

Example:

Node1 CPU usage = 80%
Node2 CPU usage = 20%
Node3 CPU usage = 40%

Scheduler prefers Node2 or Node3, not Node1.

2. Least allocated resources

Nodes with more free resources get higher score.

Example:

Node1 free CPU = 0.5
Node2 free CPU = 3
Node3 free CPU = 1

Best node:

Node2
3. Pod spreading (High availability)

Scheduler avoids placing all pods on one node.

Example deployment:

3 pods of same app

Scheduler distributes them:

Node1 → 1 pod
Node2 → 1 pod
Node3 → 1 pod

This prevents single node failure affecting all pods.

4. Image locality

If a node already has the container image downloaded, scheduler may prefer that node.

Example:

Node1 → image already exists
Node2 → image not present

Node1 may get higher score because pod can start faster.

Example scoring
Node1 → score 40
Node2 → score 85
Node3 → score 65

Best node:

Node2
3️⃣ Phase 3 — Binding (Assigning pod to node)

After scoring, scheduler sends an instruction to the API server:

Bind pod → Node2

Now Kubernetes starts the pod on Node2.

Flow:

Pod created
      ↓
Scheduler filtering
      ↓
Scheduler scoring
      ↓
Best node selected
      ↓
Pod bound to node
      ↓
Kubelet starts container
