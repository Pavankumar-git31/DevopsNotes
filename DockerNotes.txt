
MONOLITHIC :
monolithic means if an application contains n number of services (let's take the Paytm has money transactions , movie tickets , train tickets etc..) if all these services are included in to one server then it will be called monolithic architecture. every monolithic architecture having one database for all the services 


MICROSERVICES :
if an application contains n number of services (let's take the Paytm application has movies , train tickets etc..) if every service having it own individual servers it is called the microservices.
every microservice architecture has its own database for each service

MICRO:
cost is more 
maintenance is more because we have to maintain the multiple servers(this is major drawback)
individual databases (more efficient to read) 


MONO:
server performance will effect when more no of users.
only single database. (less efficient)

COMPONENTS :

DOCKER SWARM : 
high availability : deploying the app on more than one servers means using cluster.

docker swarm is container orchestration tool 
it is used to manage multiple containers on multiple nodes.
each node will have a copy of single container.
here we have manager and worker nodes.
manager node will create a containers and send to worker nodes.
worker nodes will take the container and manage them.
manager node communicates with worker node by using token.

SETUP :  
create 3 servers (1 = manager 2 = worker nodes ) (all traffic is must)
install docker and set the hostnames.
in all manager and worker nodes 

dnf install docker -y
systemctl start docker 
systemctl status docker 
hostnamectl set-hostname manager / worker-1 / worker-2

in manager node :
docker swarm init	: it is going to generate the token
copy the token to all worker nodes 
docker node ls   	: to list the all nodes that joined 

Note : if we want token again
docker swarm join-token worker
docker swarm join-token worker
now the cluster is ready.

SERVICES :
it is a way of exposing the application in docker.
with services we can create multiple copies of same container.
we can distribute the containers to all servers.

how to create service :
docker service create --name movies --replicas=3 --publish 81:80 repo_name:latest


commands: 
docker service ls	: to list the services 

docker service ps train	: to list all containers on train replicas 

docker service inspect train	: to give the complete information about that service 

docker service scale train=10	: it will scale to 10 services 

docker service rollback train	: it will roll back the previous scaling 

docker service rm train		: it is going to remove the train service , docker service ls 

Note : if you delete the container again it is going to heal the container that means the container will recreate again.

NETWORKING CONCEPTS :
suppose i have two containers app , database 
if app server wants to communicate with database server with ip addresses it is okay but some issues the we nned to delete the container and create new one in that case the ip addresses is going to change. so there is connection issues in order to overcome that one we have to create the network.
coming to networks we have 4 types of network:
1) Bridge Network :
in same server i have the two different containers if they want to communicate with each other in that case bridge network is used.

2) Host Network :
it is nothing but the ec2 server network is used by the container nothing but the container ip address and ec2 ip address is same in that case we are going to use the host network.

3) None Network : 
when you dont want to expose the container to the world in that case we are going to use the none network that means it will not provide the any network to the container.

4) Overlay Network :
used to communicate containers with each other across multiple docker hosts.
