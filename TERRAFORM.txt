TERRAFORM OUTPUTS :
here we are creating the resources so resource having lot of information like private ip address and public ip address , dns --etc. so whenever we want to print the resource information in that case we are going to use the terraform outputs 

HOW TO WRITE THE OUPTUT BLOCK :

vim main.tf

provider "aws"{
region = "ap-south-1"
}

ami = "ami-mehrgqkjwf"
type = "t2.medium"

Name = "pavan"
}
}

value = aws_instance.one.public_ip
}

multiple values :
output "pavan"{
value = [aws_instance.one.public_ip, aws_instance.one.private_ip]
}

the moment when you give the terraform apply auto-approve it will not create the new server the resource block will not execute because it is already there so it will execute the changed output clock now it will print the private and public ip addresses.

TERRAFROM WORKSPACE :
WORKSPACE : where we write the code and execute the operations.
it is used to isolate the environments.
in real time all the works we do on workspaces only.
if we perform an operation on dev workspace it wont effect another workspace.

COMMANDS :
i want to see the workspaces 	: terraform workspace list 
create a new workspace 		: terraform workspace new dev
to show the all workspaces 	: terraform workspace list
to show current workspace	: terraform workspace show
to switch the workspaces	: terraform workspace select prod
to delete the workspace 	: terraform workspace delete prod

now to switch to required workspace and need to change env in local according to that instead of that i will put the env = "${terraform.workspace}"
now it will directly take the current env name 

NOTES :
we cant delete the current workspace.
we cant delete our workspace without deleting the resources in that workspace.
we cant delete the default workspace.

THINGS NEED TO CONSIDER WHILE DEPLOYING :
-> need to take the backups of the current builds and databases.
-> need to provide the isolate environments for dev, test and prod
-> can able to roll back if the deployment fails.
-> make sure we are deploying the correct env and client application servers.


DEPLOYMENT STARTEGIES :

BLUE GREEN DEPLOYMENT :
if i have to deploy the new version i will create a new servers and deployed the app and i will detach the old servers from old server and attach new servers.

ROLLING STRATEGIES :
we are going to create new servers parallel we are removing the old servers.
old code and new code must be running on same time in parallel environment that is old environment and new environment.

CANARY DEPLOYMENT :
first i will redirect the 30 % of users to new code and then if every thing works fine then i will redirect all users to new code.

RECREATE STRATEGY :
we need to down the all servers and we need to deploy the new version then we need to bring the services in to running state.

A/B DEPLOYMENT :
we will use the two ports here one port is old code and new code is another code . first we will redirect the some users to both ports which one is working fine based on that we will redirect the whole traffic.
